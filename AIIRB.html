Project Description

AIIRB is an autonomous mobile robot built on the TI MSP432P401R LaunchPad. It is designed to explore an unknown environment by continuously scanning ahead with an ultrasonic range‑finder mounted on a servo, selecting the direction with the most open space, and driving forward until it encounters an obstacle. When a bump sensor is triggered, the robot backs up and rescans.

Simultaneously, a Raspberry Pi with an infrared camera is triggered at each decision point to capture thermal images. The images are saved for later analysis, making the platform suitable for thermal inspection, search & rescue simulations, or educational robotics projects.

Key hardware components:

  • TI MSP432P401R LaunchPad (main controller)
  • HC-SR04 Ultrasonic Sensor (rangefinding)
  • Servo (sweeps ultrasonic sensor 0°–180°)
  • Bump switches (edge collision detection)
  • Two DC motors with individual PWM control
  • Raspberry Pi 3B+ with IR camera (low-light imaging)
  • Wi‑Fi Access Point (manual live‑streaming mode)

Original Conceptualization

The project was inspired by the idea of a low‑cost, autonomous inspection robot that could navigate inside a building while capturing infrared images, useful for detecting heat leaks, overheating electrical panels, or locating people in low‑visibility conditions.

The MSP432 would handle real‑time control (motor PWM, servo position, ultrasonic timing, and bump interrupts), while a Raspberry Pi would manage the IR camera and provide Wi‑Fi connectivity for live video streaming. The two boards communicate via simple GPIO handshake signals.

The navigation strategy uses a state machine (SCAN → TURN → FORWARD → BUMP) that prefers straight movement unless a clear side path offers significantly more room, implementing hysteresis to prevent oscillation.

Navigation State Machine

    │  SCAN   │◄────────────────────│ BUMP │

      │   │          all blocked       │
      │   └───────────────────────────►│
      │                                │
      ▼                                │

    │ T90  │─────────────────────────► │
    └──────┘                           │
    ┌──────┐                           │
    │ T0   │─────────────────────────► │
    └──────┘                           │
    ┌──────┐                           │
    │ T180 │─────────────────────────► │
    └──────┘                           │

State Descriptions:

  • SCAN – Stop motors, trigger camera, sweep ultrasonic to 0°, 90°, 180°. Measure distances, apply hysteresis to choose best direction.
  • T0 / T90 / T180 – Turn to the selected angle, take a picture, then drive forward until bump interrupt.
  • BUMP – Reverse for 1 second, then return to SCAN.

Software Modules

  • Motor.c – PWM‑based motor control (Forward, Backward, Left, Right, Stop) using TimerA0.
  • BumpInt.c – Hardware interrupt handler for six bump sensors (Port4, falling‑edge). Sets leftBump, rightBump, and wasInterrupt flags.
  • Servo/PWM – TimerA3 generates 20 ms period PWM; pulse width varied to sweep ultrasonic sensor.
  • Ultrasonic (distanceInCm, pulseIn) – TimerA2 captures echo pulse width; distance calculated using speed of sound.
  • takePicture() – GPIO handshake triggers Raspberry Pi camera via Port8 pins.

Obstacles Faced

ChallengeSolution
Wi‑Fi Hotspot & Streaming Setting up the Raspberry Pi as a Wi‑Fi access point (AP) and streaming live video proved difficult. The script required precise dependencies and system configuration. Additionally, the Pi’s Python virtual environment had to be invoked with sudo using the full path to the venv Python interpreter to retain hardware access while loading the proper packages (picamera2, flask, rpi-networking).
Motor Speed Calibration The two DC motors did not rotate at the same speed under identical PWM duty cycles, causing the robot to veer off course. The issue was resolved by manually adjusting the left and right PWM duty cycles (R_DUTY = 13499, L_DUTY = 14999) were fine‑tuned after repeated straight‑line tests) until consistent forward motion was achieved.
Ultrasonic Sensor Recalibration & Port Fried The original ultrasonic sensor port (Port6) was damaged, likely due to an accidental short. The wiring was moved to Port7, requiring changes in Init_Ports.c and #define macros (TRIGGER, ECHO). The sensor also needed recalibration: the timeout value was tuned, and the formula for converting pulse width to distance was verified against known reference lengths.
State Machine Debugging Initial versions caused the robot to oscillate between forward and scan states. Adding hysteresis (HYST_CM = 5) and dead‑end detection (DEAD_END_CM = 5) prevented the robot from jittering when distances were nearly equal.

Testing and Verification

The robot was tested inside a makeshift “room” constructed from cardboard bricks. The enclosed space simulated walls and corners, allowing the ultrasonic sensor and bump switches to be evaluated in a controlled environment.

Test Procedure:

  • Power on the MSP432 and Raspberry Pi; verify Wi‑Fi AP appears (RPi_Camera).
  • Place robot in the center of the cardboard room.
  • Observe SCAN behavior: servo sweeps, distances logged via UART.
  • Verify direction selection: robot should turn toward most open space.
  • Confirm bump interrupt: robot reverses and rescans upon impact.
  • Connect to live stream using the IP address generated by the PI in manual mode.
  • Review captured thermal images stored on the Pi.

Results: The robot consistently navigated the cardboard enclosure without escaping. Hysteresis prevented unnecessary turn oscillations. Bump detection worked reliably. Wi‑Fi streaming delivered a smooth 640×480 MJPEG feed with acceptable latency.

Possible Improvements (with more time)

  • Finer angular resolution: Restore the 45° and 135° scan positions (commented out in code) to improve navigation in complex environments.
  • Wi‑Fi handshake robustness: Implement two‑way GPIO handshake so the MSP432 waits for a “ready” pulse from the Pi before moving the servo, ensuring every angle is captured.
  • Sensor fusion: Add wheel encoders for odometry and an IMU for heading estimation, enabling more precise turns and dead‑reckoning.
  • Thermal anomaly detection: Run a lightweight machine‑learning model on the Raspberry Pi to flag unusual heat signatures in real time.
  • Remote control web interface: Extend the Flask server to include buttons for manual drive and servo control, plus a live thermal overlay.
  • Battery monitoring: Add ADC reading of the battery voltage to trigger a low‑power warning before the robot stops unexpectedly.
  • Enclosure design: 3D print a protective shell that houses all electronics neatly while exposing the ultrasonic sensor and camera.
